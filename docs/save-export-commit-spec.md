# Save, Export, and Export Commit Spec

## Purpose

This document reviews the current `flo` save/export model and specifies a cleaner split between:

- `Save`: canonical workspace persistence
- `Export`: derived, shareable output
- `Commit Export`: versioned changes made inside an exported bundle

This review is based on the current implementation in:

- `src/lib/file-ops.ts`
- `src/lib/export-context.ts`
- `src/lib/context-parser.ts`
- `src/App.tsx`
- `src-tauri/src/commands/save.rs`
- `src-tauri/src/commands/load.rs`
- `src-tauri/src/commands/history.rs`
- `src-tauri/src/commands/files.rs`

## Current Behavior

### Save

The Save button and `Cmd+S` call `saveProject()` in `src/lib/file-ops.ts`.

Current flow:

1. If the workspace has no `dirPath`, the app opens a save dialog and stores the selected path as the workspace path.
2. The frontend serializes cards, edges, viewport, and a generated `context.md`.
3. The frontend calls the Tauri command `save_project_v2`.
4. The backend writes multiple files into the selected folder.
5. The frontend then writes a JSON snapshot into `history/`.

Important implementation details:

- Save is folder-backed, even though the first save uses a save dialog rather than a folder picker.
- Save writes both canonical data and generated artifacts.
- Save always regenerates `context.md`.
- Save also writes per-card markdown files for cards with docs.

### Load

Open loads a directory, not a single file.

Current load behavior:

- First tries the v2 root-level split format:
  - `meta.json`
  - `cards.json`
  - `edges.json`
  - `viewport.json`
- Falls back to legacy `.flo/canvas.json`
- Does not read per-card markdown sidecars back as source of truth

### Export

`Export for AI` and `Cmd+E` call `exportContext()` in `src/lib/file-ops.ts`.

Current export behavior:

- Generates `context.md` from in-memory workspace state
- Opens a save dialog for a single markdown file
- Writes exactly one file: `context.md`

Export does not write:

- `meta.json`
- `cards.json`
- `edges.json`
- `viewport.json`
- `history/`
- `assets/`
- per-card markdown sidecars

### External edits today

There is already a watcher on the workspace folder.

- If `context.md` changes, the app reads it and parses only the `## Structure` section back into cards and hierarchy edges.
- If `meta.json`, `cards.json`, `edges.json`, or `viewport.json` change, the app offers a full reload from disk.
- Edits to typed markdown sidecars like `projects/foo.md` are not re-imported.
- Edits to exported standalone `context.md` files outside the workspace are not tracked.

## Current On-Disk Structures

### Current Save Hierarchy

Actual v2 save output today:

```text
<workspace>/
  meta.json
  cards.json
  edges.json
  viewport.json
  context.md
  assets/
  history/
    2026-03-31T12-34-56.789Z.json
  projects/
    some-project.md
  processes/
    some-process.md
  references/
    some-reference.md
  brainstorms/
    some-brainstorm.md
```

Notes:

- `assets/` and `history/` are created on save even if empty.
- Typed card folders are only created for cards where `has_doc` is true and `doc_content` is non-empty.
- `context.md` is auto-generated during Save, not hand-authored canonically.
- Snapshot files are full JSON snapshots written after each save.

### Current Explicit Export Hierarchy

Actual export output today:

```text
<anywhere-user-chooses>/
  context.md
```

That is the entire explicit export product today.

## What Actually Gets Exported

The exported markdown is generated by `generateContextMd()`.

### Included in `context.md`

The export currently contains these sections when relevant:

```text
# <project name>
> generated summary
> goal
> agent focus

## Rules

## Structure

## Workflows

## Context References

## Agent Hints

## Detailed Documents
```

More precisely:

- Project name
- Generated summary line
- Goal line, if set
- Goal-shaped agent focus line
- Rules:
  - body lines from `project` and `process` cards
- Structure:
  - hierarchy tree
  - card type labels
  - card titles
  - first summary line
  - tags
  - optional inline agent hints
  - hidden HTML comments with card ids
- Workflows:
  - ordered flow chains from `flow` edges
  - title plus first summary line
  - optional inline agent hints
- Context References:
  - rendered from `reference` edges
  - scope-sensitive output for `title`, `summary`, `section`, `full`
- Agent Hints section:
  - only when hint export mode is `section`
- Detailed Documents:
  - converted markdown/plain text from card docs
  - only when card docs are enabled for export
  - only for top-level doc cards or cards targeted by full references

### Excluded from `context.md`

- Comments on cards
- Viewport
- History snapshots
- Imported assets as files
- Raw graph JSON
- Most per-card metadata beyond title, summary, tags, and agent hint
- Brainstorm cards by default
- Any card carrying an excluded tag

### Important nuance

`Save` writes a `context.md`, but that does not mean `context.md` is the canonical saved model.

Today the canonical load path is still:

- `meta.json`
- `cards.json`
- `edges.json`
- `viewport.json`

`context.md` is generated output that happens to live next to the canonical files.

## Current Problems

### 1. Save and Export are conceptually mixed

Current Save writes both:

- canonical state
- derived artifacts

This blurs the product model. A user cannot easily tell whether `context.md` and typed doc folders are editable source files or just generated output.

### 2. Per-card markdown sidecars are not true source of truth

The app writes typed markdown files on save, but load does not read them as canonical input.

That means:

- users can edit them in the Files tab or Finder
- the app does not round-trip those edits back into workspace state
- the files appear more authoritative than they actually are

### 3. Sidecar filenames are title-based and collision-prone

Per-card doc files are written as slugified titles inside type folders.

That creates several failure modes:

- two cards with the same title can overwrite each other
- renaming a card can leave an orphaned old file behind
- file paths are unstable across title changes

Card ids exist, but the sidecar file naming does not use them.

### 4. The existing external-edit parser is intentionally partial

`parseContextMd()` only parses the `## Structure` section back into:

- card creation/removal
- title/body/tag/agent-hint changes
- hierarchy edge rebuilds

It does not round-trip:

- workflows
- reference edges
- detailed document content
- comments
- viewport
- assets

So "editing the export" currently means "editing only a narrow structural subset".

### 5. Save path semantics are confusing

The first Save uses a save dialog with a default name like `<project>.flo`, but subsequent Open uses a directory picker. The product is treating the saved thing as a folder-backed workspace.

This should be made explicit in the UI and in product language.

### 6. `created` is being rewritten on each save

The current payload writes `created: new Date().toISOString()` on every save. That behaves more like `modified_at` than `created_at`.

The model should be:

- `created_at`: stable once
- `updated_at`: rewritten each save

## Design Alignment Correction

After re-checking the existing design docs, the folder hierarchy for the saved local workspace should follow the system's current split-format contract:

```text
<workspace>.flo/
  meta.json
  cards.json
  edges.json
  viewport.json
  context.md
  assets/
  history/
```

This is consistent with:

- `docs/superpowers/plans/2026-03-30-extended-design-impl.md`
- `docs/cloud-option-spec.md`

Important correction:

- `context.md` should remain in the saved local workspace format for now
- typed per-card markdown sidecars should not be written by the v2 save path
- the export bundle hierarchy should be additive and separate from the local workspace format

So the correct near-term model is:

- local workspace save format keeps `context.md`
- local workspace save format stops creating typed sidecar folders
- export bundles introduce the git-like commit structure without changing how normal workspaces are opened

## Product Model Recommendation

### Save

`Save` should mean:

- persist the canonical workspace
- preserve full-fidelity round-trip
- be optimized for app correctness, not human browsing

Save should not be the main place where export artifacts are defined.

### Export

`Export` should mean:

- produce a derived representation for AI, sharing, or external editing
- be reproducible from a workspace save plus export options
- be allowed to omit data that is irrelevant to the target use case

### Commit Export

`Commit Export` should mean:

- record a versioned change inside an exported bundle
- preserve a changelog of edits made after export
- optionally pull accepted export changes back into the workspace

This should be treated as export-level versioning, not workspace canonical history.

## Proposed Future File Hierarchies

### A. Canonical Save Hierarchy

Recommended save structure:

```text
<workspace>.flo/
  meta.json
  cards.json
  edges.json
  viewport.json
  context.md
  assets/
  history/
    2026-03-31T12-34-56.789Z.json
```

Recommended metadata fields:

```json
{
  "workspace_id": "uuid",
  "name": "My Workspace",
  "created_at": "2026-03-31T12:00:00.000Z",
  "updated_at": "2026-03-31T12:34:56.789Z",
  "format_version": 3,
  "goal": "..."
}
```

Recommendation:

- keep writing `context.md` as part of the local workspace save format for compatibility with the current design
- stop writing typed card markdown folders as part of Save
- treat `context.md` as generated within the workspace, not canonical source of truth
- keep any future additional export artifacts outside the normal workspace root, for example in explicit export bundles

### B. Explicit Export Bundle Hierarchy

Recommended export structure:

```text
<export-name>.flo-export/
  context.md
  export.json
  manifest.json
  project/
    project.md
    export-options.json
  cards/
    project/
      <card-id>--<slug>.md
    process/
      <card-id>--<slug>.md
    reference/
      <card-id>--<slug>.md
    brainstorm/
      <card-id>--<slug>.md
  graph/
    hierarchy.json
    flows.json
    references.json
  assets/
    ...
  .flo-export/
    HEAD
    commits/
      000001-initial/
        commit.json
        manifest.json
      000002-followup/
        commit.json
        manifest.json
```

Why this structure:

- `context.md` remains the AI-friendly aggregate
- per-card files make edits and diffs human-readable
- graph files make relationship changes explicit
- `manifest.json` makes change detection deterministic
- `.flo-export/commits/` provides git-like history without implementing actual git

## Export File Roles

### Canonical within the export bundle

These files are the editable tracked source for the export bundle:

- `project/project.md`
- `project/export-options.json`
- `cards/**/*.md`
- `graph/*.json`

### Derived within the export bundle

These files are generated from the export bundle source:

- `context.md`
- `manifest.json`

Recommendation:

- treat `context.md` as generated and commit-tracked, but not the primary editable source
- avoid using monolithic `context.md` as the only editable artifact for round-trip changes

## Card File Spec

Recommended per-card file format:

```md
---
card_id: 8f2d6e7d-...
type: process
title: Onboard New User
tags:
  - activation
agent_hint: Emphasize first-session friction.
include_in_export: true
---

## Summary

Short card body / summary.

## Document

Full card document markdown here.
```

Why:

- stable `card_id` eliminates slug collisions
- title can change without changing identity
- body and doc content are separated cleanly
- export parser can round-trip more than just structure

## Export Commit Spec

### Goals

- Let users continue editing an exported bundle after initial export.
- Let the bundle keep its own revision history.
- Let flo detect what changed and show a reviewable diff.
- Let users optionally merge those changes back into the workspace.

### Export identity

Each export bundle needs:

- `export_id`
- `workspace_id`
- `base_workspace_revision`
- `export_profile`
- `created_at`

Store this in `export.json`.

Example:

```json
{
  "export_id": "uuid",
  "workspace_id": "uuid",
  "base_workspace_revision": "save:2026-03-31T12:34:56.789Z",
  "export_profile": "ai-handoff-v1",
  "created_at": "2026-03-31T12:35:10.000Z"
}
```

### Commit metadata

Each commit should record:

- `commit_id`
- `parent_commit_id`
- `message`
- `author`
- `created_at`
- `source`
- `changed_files`
- `workspace_merge_status`

Example:

```json
{
  "commit_id": "000002-followup",
  "parent_commit_id": "000001-initial",
  "message": "Clarify onboarding workflow and tighten export tags",
  "author": "local-user",
  "created_at": "2026-03-31T13:02:00.000Z",
  "source": "external-edit",
  "changed_files": [
    "cards/process/8f2d6e7d--onboard-new-user.md",
    "graph/flows.json",
    "context.md"
  ],
  "workspace_merge_status": "not_applied"
}
```

### HEAD model

The export bundle should have:

- `.flo-export/HEAD`
- `.flo-export/commits/<commit-id>/commit.json`
- `.flo-export/commits/<commit-id>/manifest.json`

This is enough to support:

- show current export head
- diff working tree vs last commit
- browse commit history
- restore an earlier export commit

### Commit flow

Recommended flow:

1. User creates an export bundle.
2. flo writes initial bundle files and `000001-initial`.
3. User or agent edits files in the export bundle.
4. flo detects dirty exported files by hashing against `manifest.json`.
5. User opens `Review Export Changes`.
6. flo shows file diff plus parsed semantic diff.
7. User enters a commit message.
8. flo writes a new commit folder and updates `HEAD`.
9. User optionally chooses `Apply commit to workspace`.

### Merge-back flow

Recommended actions for an export commit:

- `Commit Only`
- `Commit and Apply to Workspace`
- `Discard Working Changes`

When applying to the workspace:

- parse per-card markdown files back into card bodies and docs
- parse graph files back into hierarchy, flow, and reference edges
- regenerate in-memory `context.md`
- create a normal workspace save and snapshot
- mark the export commit as `applied`

### Conflict rules

If the workspace changed after the export base revision:

- compare current workspace revision with `base_workspace_revision`
- if different, show a 3-way merge prompt
- allow per-file or per-card accept/reject
- never silently overwrite canonical workspace state

## Required Product Decisions

### Decision 1: Is `context.md` editable source or generated output?

Recommendation:

- generated output

Reason:

- a single monolithic file is poor for granular diffs
- current parser already proves only a subset can be round-tripped safely
- per-card files and graph files are a much better commit surface

### Decision 2: Should Save continue writing export artifacts?

Recommendation:

- keep writing `context.md`, but no additional export bundle artifacts in normal Save

Reason:

- the current system design already includes `context.md` in the local workspace format
- `context.md` is also used by the existing watcher, ghost preview, and external-edit flow
- explicit export bundles should stay separate so the normal workspace hierarchy remains stable

### Decision 3: Should export commits reuse git directly?

Recommendation:

- not initially

Reason:

- a git-like model is useful
- actual git integration adds credential, binary, and UX complexity
- a lightweight internal commit store delivers most of the product value first

## Implementation Plan

### Phase 1: Clarify current save model

- Add visible workspace path and `Save As`
- Rename metadata fields to `created_at` and `updated_at`
- Stop presenting generated files as canonical source

### Phase 2: Separate export from save

- Remove typed card markdown sidecars from default Save output
- Add `Export Bundle...` command that writes the hierarchical export format

### Phase 3: Add export tracking

- Add `export.json`
- Add `manifest.json`
- Add `.flo-export/HEAD`
- Add initial export commit creation

### Phase 4: Add commit workflow

- Detect working-tree changes in export bundle
- Add review and commit UI
- Write commit metadata and file manifests

### Phase 5: Add merge-back

- Parse card markdown and graph files back into workspace changes
- Add conflict handling against current workspace head
- Save merged result into normal workspace history

## Acceptance Criteria

### Save

- Save writes the designed local workspace format: canonical JSON state, generated `context.md`, assets, and workspace history.
- Load round-trips all canonical workspace fields without consulting export artifacts.
- `Save As` and `Open` both clearly operate on a workspace folder.

### Export

- Export produces a folder hierarchy, not just a single markdown file.
- Export includes both human-editable card files and generated aggregate `context.md`.
- Export options are recorded in the bundle.

### Commit Export

- An export bundle can be reopened and diffed against its last export commit.
- Users can create a named commit after external edits.
- Users can browse export commit history.
- Users can apply a committed export change back into the canonical workspace.
- Conflicts are explicit and reviewable.

## Bottom Line

Today:

- `Save` is the real persistence path
- `Export` is only a single generated `context.md`
- `context.md` is partially re-importable, but only for structure edits

Recommended future state:

- `Save` owns canonical workspace persistence
- `Export` produces a tracked folder hierarchy for AI and external editing
- `Commit Export` adds git-like version history to exported bundles without making exports the canonical source of truth
